# knowledge_rag：项目启动方法与实现进度

更新时间：2026-02-23（以本地仓库状态为准）

## 1. 项目概述

本项目是面向“工业知识库混合检索”的工程原型，当前已完成：
- 基础设施（Elasticsearch + Qdrant + Kibana）容器化部署
- 文本切片数据（JSONL）准备
- 离线入库：JSONL → Qdrant（向量检索）与 Elasticsearch（关键词检索）
- 检索验证脚本：分别验证 ES BM25 与 Qdrant 向量检索
- 模型服务（FastAPI）：`/health` 与 `/embed`（为 Java 检索平台提供向量化能力）

尚未完成（待落地）：
- 精排（Cross-Encoder reranker）推理服务与微调训练流程
- Java 检索平台后端（Spring Boot）：ES + Qdrant + RRF，并调用模型服务 `/embed`

## 5. 数据与入库

### 5.1 数据格式（JSONL）
当前切片数据默认位置：
- `data/processed/knowledge.jsonl`

每行是一条切片记录（示例字段）：
- `chunk_id`：切片唯一标识
- `content`：切片文本
- `source` / `source_url`：来源
- `page_num`：页码/行号（用于回溯定位）
- `anchor_text`：锚点文本（用于前端页内搜索/高亮）

### 5.2 全量入库（推荐：Qdrant + ES 同步）

脚本：`experiments/scripts/offline_indexing.py`

```powershell
# 全量入库：重建 Qdrant collection，并同步写入 ES index
conda run -n rag_env python experiments/scripts/offline_indexing.py \
  --input data/processed/knowledge.jsonl \
  --qdrant-url http://localhost:6333 \
  --collection knowledge_chunks \
  --model BAAI/bge-base-zh-v1.5 \
  --batch-size 64 \
  --recreate \
  --with-es \
  --es-url http://localhost:9200 \
  --es-index knowledge_chunks
```

说明：
- 当前 JSONL 不包含向量字段时，脚本会在入库阶段使用 `SentenceTransformer` 现场生成向量，因此入库会更慢但流程更简单。
- 若你已提前把向量写回 JSONL（例如字段名 `embedding`），可用 `--vector-field embedding` 跳过向量生成。

### 5.3 入库结果验证
```powershell
# Qdrant points 数量
conda run -n rag_env python -c "from qdrant_client import QdrantClient; c=QdrantClient(url='http://localhost:6333'); print(c.get_collection('knowledge_chunks').points_count)"

# ES docs 数量
conda run -n rag_env python -c "from elasticsearch import Elasticsearch; es=Elasticsearch('http://localhost:9200'); print(es.count(index='knowledge_chunks')['count'])"
```

## 6. 检索验证（离线脚本）

脚本：`experiments/scripts/05_test_search_engines.py`

功能：
- Elasticsearch：对 `content` 字段做 match（BM25）
- Qdrant：对同一 query 做向量检索（TopK）

运行：
```powershell
conda run -n rag_env python experiments/scripts/05_test_search_engines.py
```

重要：向量模型必须一致
- Qdrant collection 的向量维度由入库时的 embedding 模型决定。
- 当前默认入库模型 `BAAI/bge-base-zh-v1.5`（768 维），测试脚本也默认使用该模型。
- 若你改用 `all-MiniLM-L6-v2`（384 维）或 `BAAI/bge-small-zh-v1.5`（512 维），需要用同一模型重新入库（`--recreate`）。

### 7.1 已实现
- 基础设施：`docker-compose.yml`
  - ES 8.11.1（关闭安全验证，便于毕设开发）
  - Qdrant latest（向量库）

- 数据：`data/processed/knowledge.jsonl`
- 入库脚本：`experiments/scripts/offline_indexing.py`
  - JSONL → Qdrant（建库 + upsert）
  - 可选：同步写入 ES（bulk）
- 检索测试：`experiments/scripts/05_test_search_engines.py`
  - ES BM25
  - Qdrant 向量检索（使用 `query_points`，兼容当前 qdrant-client）
- 模型服务：`model_service/main.py`
  - `GET /health`：模型服务状态与配置
  - `POST /embed`：批量生成 query/text 的向量（默认 768 维）

### 7.2 未完成（下一阶段）
- 精排服务与模型微调：`experiments/scripts/fine_tune_reranker.py` 为占位
  - 建议拆分：训练脚本（Colab/远端）+ 推理服务（本地 FastAPI/独立服务）

模型服务启动（在项目根目录执行）：
```powershell
uvicorn model_service.main:app --reload --port 8001
```
Swagger：`http://127.0.0.1:8001/docs`

接口调用示例：
```powershell
# health
curl.exe http://127.0.0.1:8001/health

# embed（批量）
curl.exe -X POST http://127.0.0.1:8001/embed -H "Content-Type: application/json" -d "{\"texts\":[\"设备 故障 处理\",\"液压 系统 漏油\"]}"
```

Java 侧接入方式（思路）：
- Spring Boot 收到 query 后，先调用模型服务 `/embed` 得到向量
- 再用该向量去 Qdrant 做向量召回，同时 ES 做 BM25 召回
- 最后在 Java 里做 RRF 融合与（可选）精排

## 8. 常见问题排查

### 8.1 Kibana 端口启动失败（bind 5601 权限错误）
现已通过映射 `15601:5601` 规避 Windows 保留端口问题。

### 8.2 ES 连接报 compatible-with=9
确保 Python 端安装：`elasticsearch==8.11.1`。

### 8.3 Qdrant 查询报向量维度不匹配
入库与查询必须使用同一个 embedding 模型（保证向量维度一致）。
